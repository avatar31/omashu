# Code Review: Omashu Codebase Improvements

> **Reviewed:** May 31, 2026  
> **Scope:** All source files — `tso.go`, `txn.go`, `node.go`, `distributed.go`, `fsm.go`, `transport.go`, `replication.go`, `config.go`, `store.go`, `badger.go`

---

## P0 — Blocking / Correctness

| # | File | Issue | Fix |
|---|------|-------|-----|
| 1 | `node.go` / `distributed.go` | **`proposeHandler` signals completion before Raft commit.** `proposeRespNotifier <- propose{ErrResp: nil}` is sent immediately after `n.Propose()` returns, which only means the proposal was *queued* — not committed. `listenProposeResponses` routes this to `proposeAndWait`, which returns `nil` while the entry is still in-flight. Callers believe the write is durable before BadgerDB has applied it. | In `proposeHandler`, remove the early `proposeRespNotifier` send. Instead, in `applyNormalEntry` (after `fsm.Apply()` succeeds), decode the command ID and route the commit result through `proposeRespNotifier`. This way the caller only unblocks when the write is actually applied to BadgerDB. |
| 2 | `tso.go` / `txn.go` | **`DoneCommit` called before FSM applies the entry.** `Txn.Commit()` calls `tso.DoneCommit(commitTs)` via `defer` before `proposeAndWait` even queues the proposal. This advances `txnMark`, allowing subsequent `ReadTs()` calls to return immediately without waiting. A reader can then read from BadgerDB at a timestamp that has been "committed" by the TSO but not yet applied by the FSM — returning stale data. | `DoneCommit(commitTs)` must be called from `applyNormalEntry` / `applyTransaction` in `fsm.go` *after* the write is persisted to BadgerDB. The FSM should notify the TSO watermark, not the proposal layer. |
| 3 | `distributed.go` | **`proposeAndWait` timeout races with `listenProposeResponses` — panic on closed channel.** On timeout, `proposeAndWait` calls `s.proposals.Delete(cmd.Id)` then `close(errCh)`. If `listenProposeResponses` has already loaded the channel from `proposals` (before the Delete) but hasn't sent yet, it will call `errCh <- r.ErrResp` on a closed channel, causing a **panic**. | Remove the `close(errCh)` calls. Keep the channel open and drain it with a `select { case <-errCh: default: }` or use a `sync.Once` flag. Alternatively, switch to a model where the FSM signals completion (per fix for #1), and `listenProposeResponses` is no longer needed for success signalling. |
| 4 | `distributed.go` / `node.go` | **Concurrent readers consume each other's `ReadState`.** `readStatesNotifier` is a single channel (capacity 2). Multiple goroutines calling `Get` / `Exists` concurrently each block in `WaitForReadState`, which takes the first available `ReadState` regardless of which `ReadIndex` request it corresponds to. Reader A may consume Reader B's `ReadState` with a lower confirmed index than A actually needs, silently skipping the wait and reading stale data. | Match `ReadState` to requests by tracking the `RequestCtx` field (which echoes the `rctx` passed to `n.ReadIndex`). Maintain a `map[string]chan raft.ReadState]` keyed by request context, and dispatch in `respondToReadStates` to the right waiter. |
| 5 | `node.go` | **`respondToReadStates` sends to `readStatesNotifier` without a `select/default`, blocking `readyHandler`.** The send `node.readStatesNotifier <- readStates[...]` is unbounded-blocking. With capacity-2 and ≥3 concurrent slow readers, `readyHandler` stalls mid-loop. This prevents `Tick()` from running, causing election timeouts and unexpected leadership changes. | Use a non-blocking send with `select { case node.readStatesNotifier <- rs: default: }`, and re-queue undelivered states back to pending for the next `processReady` iteration. |
| 6 | `distributed.go` | **`tso.StartServing` is called synchronously inside `onLeaderChange`, blocking `readyHandler`.** `onLeaderChange` is invoked from `handleLeaderChange` → `processReady`, all in the same `readyHandler` goroutine. `StartServing` performs a BadgerDB read (`getVal`) which can be slow. During this time, Raft ticks do not fire and heartbeats stall — the node may immediately lose the election it just won. | Wrap `StartServing` and the leader-gain path in a `go func()` so `readyHandler` remains unblocked. Ensure TSO initialization is complete before accepting writes (e.g., set a `serving` atomic flag). |
| 7 | `tso.go` | **`tso.readers` is a set, not a reference counter — multiple transactions with the same `readTs` cause the `readMark` watermark to stall.** `currentReadTs()` returns `current.Compose()` *without* incrementing the logical counter. Concurrent `BeginTxn` calls between two commits return the same `readTs`. Both call `readMark.Begin(readTs)`, but `doneRead` removes the map entry on the first `Discard`, so the second `Discard` skips `readMark.Done`. The watermark is permanently stuck, `cleanupCommittedTransactions` never fires, and `committedTxns` grows unboundedly. | Change `readers map[uint64]struct{}` to `readers map[uint64]int` (reference counter). Increment on `ReadTs`, decrement on `doneRead`, call `readMark.Done` only when the count reaches 0. |
| 8 | `transport.go` | **`Transport.Start` continues executing after sending an error to `errChan`.** After each `node.errChan <- err`, execution falls through to subsequent code (peer loop, `http.Server` setup, `ListenAndServe`). A failure in `tr.raftTr.Start()` or `url.Parse` causes a cascade of further panics / nil-dereferences. | Add `return` after every `node.errChan <- err` in `Start`. Since `Start` has no return value, consider changing it to `func Start(...) error` and propagating errors directly to the caller. |

## P1 — Important / Should Fix

| # | File | Issue | Fix |
|---|------|-------|-----|
| 9 | `distributed.go` | **Blocking send to `ProposeReqNotifier` before the `DefaultProposeTimeout` timer starts.** `s.node.ProposeReqNotifier() <- propose{...}` happens before `time.After(DefaultProposeTimeout)`. If `proposeHandler` is blocked or slow, this send blocks indefinitely. The caller can wait for far longer than `DefaultProposeTimeout` with no recourse. | Wrap the channel send in a `select` with `time.After(DefaultProposeTimeout)` so the total operation — including the queuing step — is bounded. |
| 10 | `tso.go` | **`allocate()` spin-loops hold `tso.mu`, starving all other TSO operations.** Two busy-wait loops (upper-bound enforcement and logical-overflow handling) call `time.Sleep(time.Millisecond/2)` while holding `tso.mu`. `DoneRead`, `newCommitTs`, and `ReadTs` all acquire `tso.mu` and will block for the full spin duration. With no timeout, a stalled `windowUpdator` causes an infinite spin. | Release `tso.mu` before spinning (copy needed fields, reacquire to check/update). Add a deadline (e.g., 2× `defaultUpperBoundWindow`) after which `allocate` returns an error rather than spinning forever. |
| 11 | `node.go` | **`proposeHandler` treats a single `n.Propose` failure as fatal.** On error, it sends to `errChan` and exits. `readyHandler` then shuts down the node. Any transient network hiccup or context deadline causes the entire Raft node to terminate. Remaining callers in `proposeAndWait` timeout silently with no actionable error. | Distinguish transient errors (e.g., `context.DeadlineExceeded`, `raft.ErrProposalDropped`) from fatal ones. Retry transient errors up to `RetryCount` times. Only send to `errChan` for unrecoverable failures. Route per-proposal errors back to the specific waiting caller. |
| 12 | `tso.go` | **`hasConflict` is O(N × M)** where N = committed transactions since the oldest in-flight read, M = reads in the current transaction. Under high write concurrency this becomes a hot critical-section inside `tso.mu`. | Replace `committedTxns []committedTxn` with a `map[string]uint64` (key → latest commitTs). Conflict check becomes O(M) lookups. Clean up stale entries in `cleanupCommittedTransactions`. |
| 13 | `tso.go` | **`TSO.Close` can double-close `stopNotifier`, causing a panic.** `Close()` is called from both `onLeaderChange` (on leadership loss) and `DistributedBadger.Close`. If both are invoked in quick succession, `close(tso.stopNotifier)` runs twice. | Use a `sync.Once` in `Close` to guarantee the channel is closed exactly once, or gate with an atomic `closing` flag. |
| 14 | `node.go` | **`Stop` does not signal `stopNotifier`, leaving `readyHandler` / `proposeHandler` goroutines to exit only via context cancellation.** If the caller's context is long-lived (e.g., `context.Background()`), the goroutines stay alive after `Stop` returns, accessing a stopped `raft.Node`. | Send to `node.stopNotifier` (twice, one per goroutine) at the start of `Stop`, before calling `node.n.Stop()`. |
| 15 | `tso.go` | **`UpdateWindow` has a double-update race: two goroutines can both pass the `upperBound <= saved` check and persist the same value twice.** Between reading `saved` and updating it, another `windowUpdator` tick or a concurrent `persistUpperboundOnStartup` call can run the same path. Not a correctness issue, but wastes a Raft round-trip. | Hold `tso.mu` for the full check-and-update sequence, or use a `sync.Mutex` dedicated to the window update path. |
| 16 | `transport.go` | **`rafthttp.Transport.Logger` is commented out.** All internal rafthttp errors (connection failures, snapshot send errors, peer timeouts) are silently discarded. This makes diagnosing peer connectivity issues extremely difficult in production. | Implement `go.etcd.io/etcd/client/pkg/v3/logutil.ZapLogger` adapter and assign it to `tr.raftTr.Logger`. |
| 17 | `fsm.go` | **`applyTransaction` duplicates the full command-type dispatch from `Apply`.** The switch-case in `applyTransaction` re-implements the same logic as the top-level `Apply` switch (SET, UPDATE, DELETE, INCR\_BY, DECR\_BY) but inline, omitting DELETE\_BY\_PREFIX entirely. Any new command type must be added in both places. | Refactor: have `applyTransaction` call `fsm.applySubCommand(ctx, txn, subCmd)` for each entry, where `applySubCommand` is a shared helper used by both `Apply` (with a single-op transaction wrapper) and `applyTransaction`. |

## P2 — Low / Nice to Have

| # | File | Issue | Fix |
|---|------|-------|-----|
| 18 | `fsm.go` | **Unchecked type assertions in `applyUpdate` and `applyTransaction`.** `msg, _ := v.(proto.Message)` silently produces a nil `msg` if `UnmarshalUpdateDelta` returns a non-proto type. `UpdateProtobufWithTxn(ctx, txn, key, nil, ...)` then stores a nil-marshalled protobuf silently. Same for the `map[string]any` JSON path. | Check the assertion: `msg, ok := v.(proto.Message); if !ok { return fmt.Errorf(...) }`. |
| 19 | `config.go` | **`PeerDialTimeout` panics with a nil-dereference if `Logger` is not yet set.** `c.Logger.Panic(...)` is called to guard against a nil `RaftConfig`, but if `validate()` hasn't run yet (and `cfg.Logger` is nil), this produces an opaque nil pointer panic rather than a helpful message. | Check `c.Logger != nil` before using it; fall back to `log.Panicf` or return a sentinel error. |
| 20 | `config.go` | **`validate` requires a non-empty `BaseDir` even for in-memory mode.** The `BaseDir == ""` check fires before the `InMemory` branch clears `BaseDir`. Users must supply a dummy path like `"/tmp/ignored"` to pass validation when using `BadgerOptions.WithInMemory(true)`. | Reorder the validation: check `InMemory` first and skip the `BaseDir == ""` guard when it is set. |
| 21 | `node.go` | **`fmt.Sprintf` used inside `zap.Info` / `zap.Panic` calls, defeating structured logging.** `node.log.Info(fmt.Sprintf("Applied %d entries..."))` produces an unindexed string field instead of separate numeric fields queryable by log aggregators. | Replace with: `node.log.Info("Applied entries", zap.Int("published", publishedEntries), zap.Int("total", len(entries)))`. |
| 22 | `node.go` | **Large blocks of commented-out code** (`committedIndex`, `term` atomic fields and their accessors) are left in-tree, inflating the file and suggesting incomplete implementation to readers. | Remove the commented-out code entirely if not needed; if the fields are planned, add a `// TODO: P2: implement <reason>` stub instead. |
| 23 | `distributed.go` | **No retry on `badger.ErrConflict` from `NewTransaction`.** When TSO conflict detection triggers, the caller receives the error but has no built-in retry. This forces every caller to implement their own retry loop, and the current proposal pipeline has no backoff strategy. | Add an optional retry loop (up to `RetryCount` attempts with exponential backoff) inside `NewTransaction` and `proposeTxnSubCommand` for `badger.ErrConflict`, as noted in the P1 TODO list. |
| 24 | `badger.go` | **`validate()` mutates the caller's `Config` struct** (`cfg.BadgerOptions = cfg.BadgerOptions.WithDir(...)`, `cfg.BaseDir = ""`). If a `Config` is reused or shared across goroutines, these side-effects cause unexpected behaviour on the second call. | Either document `Config` as single-use after `NewBadger`/`NewDistributedBadger`, or deep-copy the options before mutating. |
